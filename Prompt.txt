MONGODB SCHEMA — Order
─────────────────────────────────────
{
  orderId      : String, unique, auto-generated (e.g., "ORD-20240601-0001")
  customerName : String, required, trim
  phone        : String, required, 10-digit validation
  garments     : [
    {
      type     : String, enum: ['Shirt','Pants','Saree','Jacket','Kurta','Bedsheet','Other']
      quantity : Number, min 1
      pricePerItem : Number   ← pulled from pricing config
      subtotal : Number       ← quantity * pricePerItem (computed before save)
    }
  ]
  totalAmount       : Number   ← sum of all garment subtotals
  status            : String, enum: ['RECEIVED','PROCESSING','READY','DELIVERED'], default 'RECEIVED'
  estimatedDelivery : Date     ← auto set to createdAt + 3 days
  createdAt         : Date, default Date.now
  updatedAt         : Date
}

─────────────────────────────────────
PRICING CONFIG — /utils/pricing.js
─────────────────────────────────────
Export a plain object:
{
  Shirt    : 50,
  Pants    : 60,
  Saree    : 120,
  Jacket   : 150,
  Kurta    : 70,
  Bedsheet : 100,
  Other    : 80
}
When an order is created, look up each garment type in this config,
set pricePerItem automatically, compute subtotal and totalAmount.
The frontend should NOT send prices — only garment type and quantity.

─────────────────────────────────────
BACKEND — ALL API ENDPOINTS
─────────────────────────────────────

Orders
  POST   /api/orders
    Body: { customerName, phone, garments: [{type, quantity}] }
    Response: { success, data: order, message: "Order created" }

  GET    /api/orders
    Query params (all optional):
      status        → filter by status
      name          → case-insensitive partial match on customerName
      phone         → exact match
      garmentType   → filter orders containing this garment type
    Response: { success, data: [orders] }

  GET    /api/orders/:orderId
    Response: { success, data: order }

  PATCH  /api/orders/:orderId/status
    Body: { status }  ← must be valid enum value
    Validates forward-only transitions:
      RECEIVED → PROCESSING → READY → DELIVERED
    Response: { success, data: updatedOrder }

Dashboard
  GET    /api/dashboard
    Response:
    {
      success: true,
      data: {
        totalOrders    : Number,
        totalRevenue   : Number,
        ordersByStatus : { RECEIVED: N, PROCESSING: N, READY: N, DELIVERED: N },
        recentOrders   : [last 5 orders, newest first]
      }
    }

IMPLEMENTATION RULES
─────────────────────────────────────
1. generateOrderId.js: format = "ORD-YYYYMMDD-XXXX" where XXXX is a
   zero-padded daily counter. Use a counter stored in DB or use
   Date.now() suffix if simpler — just keep it unique and readable.

2. All API responses must follow: { success: Boolean, data: any, message: String }

3. errorHandler.js middleware must catch all async errors and return:
   { success: false, message: error.message } with appropriate HTTP status.

4. Use express-async-errors or wrap controllers in try/catch.

5. Status transition validation: reject any PATCH request that tries to
   skip a step or go backwards. Return 400 with a clear message.

6. estimatedDelivery = createdAt + 3 days. Store in DB. Display on frontend
   formatted as "DD MMM YYYY" (e.g., "05 Jun 2024").

7. CORS: allow all origins in development. Use cors npm package.

8. MongoDB connection in db.js: read URI from process.env.MONGO_URI.
   Log "MongoDB connected" on success.

9. Frontend Axios instance: base URL from import.meta.env.VITE_API_URL.
   Add response interceptor to extract data.data automatically.

10. StatusBadge component: color-coded pill.
    RECEIVED = gray, PROCESSING = blue, READY = green, DELIVERED = dark green.

11. Live bill preview on Create Order page: use React state to recompute
    total whenever garment type or quantity changes. Import pricing from
    a frontend copy of the pricing config at /src/utils/pricing.js.

12. Add seed script at backend/seed.js that inserts 10 sample orders
    with mixed statuses, garment types, and customers. Run with:
    node seed.js
